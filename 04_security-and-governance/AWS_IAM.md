# AWS IAM（Identity and Access Management）

## IAMポリシー（Policy）

**「何を許可・拒否するか」を定義したルール**。JSON形式で記述する。

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

### ポリシーの種類

| 種類 | 説明 |
|------|------|
| **AWS管理ポリシー** | AWSが用意済みの汎用ポリシー（例: `AmazonS3ReadOnlyAccess`） |
| **カスタマー管理ポリシー** | 自分で作成・管理するポリシー |
| **インラインポリシー** | ユーザー・グループ・ロールに直接埋め込むポリシー（再利用不可） |

### ポリシーの適用先

ポリシー単体では何もしない。必ず**誰か（何か）にアタッチ**して使う。

```
IAMポリシー → アタッチ → IAMユーザー
                       → IAMグループ
                       → IAMロール
```

## IAMロール（Role）

**「誰かが一時的に引き受ける権限のセット」**。パスワードやアクセスキーを持たず、
一時的な認証情報（STSトークン）を発行して使う。

### ロールの使いどころ

| ユースケース | 説明 |
|------------|------|
| **AWSサービスへの権限付与** | EC2やLambdaがS3やDynamoDBにアクセスする |
| **クロスアカウントアクセス** | 別のAWSアカウントのリソースにアクセスする |
| **フェデレーション** | 社内のID（Google, Active Directory等）でAWSにアクセスする |

### AssumeRole とは

**ロールを「引き受ける」アクションのこと**。ロール自体の名前ではない。

| 用語 | 意味 |
|------|------|
| **IAMロール** | 権限をまとめたバッジ（モノ） |
| **AssumeRole** | そのバッジを引き受けるアクション（動作） |

```
Glueジョブ起動
    ↓
GlueがIAMロールをAssume（引き受ける）← これがAssumeRole
    ↓
AWS STSが一時トークンを発行
    ↓
そのトークンでS3にアクセス
```

「ロール ＝ AssumeRole」ではなく、「ロールをAssumeする操作 ＝ AssumeRole」。

### ロールの構成要素

```
IAMロール
├── 信頼ポリシー（Trust Policy）
│     誰がこのロールを引き受けられるか（例: Lambda, EC2, 別アカウント）
└── アクセス許可ポリシー（Permission Policy）
      このロールで何ができるか（IAMポリシーをアタッチ）
```

---

## IAMポリシー vs IAMロール

| 観点 | IAMポリシー | IAMロール |
|------|------------|----------|
| 役割 | 権限のルールを定義する | 権限をまとめて引き受ける主体 |
| 認証情報 | 持たない | 一時的なSTSトークンを発行 |
| 適用対象 | ユーザー・グループ・ロールにアタッチ | AWSサービス・ユーザー・外部IDが引き受ける |
| 単体での効力 | なし（アタッチしないと意味がない） | あり（ポリシーをアタッチすれば使える） |

### アナロジーで覚える

```
IAMポリシー ＝ 「入館できる場所のリスト」（ルール書）
IAMロール   ＝ 「その入館証を持つ役職バッジ」（バッジを渡されたら入れる）
```

## 実務での典型パターン

### LambdaがS3にアクセスする

```
Lambda関数
    ↓ 引き受ける（AssumeRole）
IAMロール（例: LambdaExecutionRole）
    ↓ アタッチされている
IAMポリシー（例: s3:PutObject を Allow）
    ↓
S3バケットへのアクセスが可能に
```

### クロスアカウントアクセス

```
アカウントA（開発）
    ↓ AssumeRole
アカウントB（本番）のIAMロール
    ↓
アカウントBのリソースにアクセス
```

---

## 最小権限の原則（Least Privilege）

必要最低限の権限だけを付与する。DEA試験でも重要な考え方。

- 広すぎる権限（`"Action": "*"`）は避ける
- 使わない権限は削除する
- 定期的に権限を見直す（IAM Access Advisor で確認可能）

---

## 試験のポイント

- **LambdaやGlueにS3アクセスを与える** → IAMロールをアタッチ（アクセスキーは使わない）
- **ポリシーの優先順位** → 明示的なDeny > Allow（Denyが勝つ）
- **クロスアカウント** → ロールの信頼ポリシーで許可 + アクセス許可ポリシーで権限付与
- **IAMユーザーにはアクセスキーを極力使わない** → ロールで代替するのがベストプラクティス
