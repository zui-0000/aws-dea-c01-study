# ストリーミング処理とバッチ処理

## バッチ処理（Batch Processing）

データを一定期間**溜めてからまとめて処理**する方式。

```
データ発生 → 蓄積 → 蓄積 → 蓄積 → [まとめて処理] → 結果
                                     ↑
                               夜間バッチなど
```

### 特徴

| 項目 | 内容 |
|------|------|
| レイテンシ | 高い（分〜時間〜日単位） |
| スループット | 高い（大量データを効率処理） |
| コスト | 比較的安い |
| 複雑さ | シンプル |

### ユースケース

- 月次・日次の売上集計レポート
- 大規模なデータウェアハウスへのETL
- 機械学習モデルの定期的な再トレーニング

### AWSサービス

- **AWS Glue** → サーバーレスETL
- **Amazon EMR** → 大規模Spark/Hadoopジョブ
- **AWS Batch** → バッチコンピューティング
- **Amazon Redshift** → データウェアハウスへの一括ロード

---

## ストリーミング処理（Streaming Processing）

データが発生した**瞬間にリアルタイムで処理**する方式。

```
データ発生 → [即時処理] → 結果
データ発生 → [即時処理] → 結果
データ発生 → [即時処理] → 結果
```

### 特徴

| 項目 | 内容 |
|------|------|
| レイテンシ | 低い（ミリ秒〜秒単位） |
| スループット | バッチより低い |
| コスト | 比較的高い |
| 複雑さ | 複雑（状態管理が必要） |

### ユースケース

- リアルタイム不正検知
- IoTセンサーデータの監視・アラート
- ライブダッシュボード
- クリックストリーム分析

### AWSサービス

- **Amazon Kinesis Data Streams** → リアルタイムデータストリーム
- **Amazon Kinesis Data Firehose** → ストリームデータの配信
- **Amazon MSK (Kafka)** → マネージドKafka
- **AWS Lambda** → ストリームのイベント処理

---

## バッチ vs ストリーミング 比較

| 観点 | バッチ | ストリーミング |
|------|--------|--------------|
| データ鮮度 | 低い | 高い |
| レイテンシ | 高い | 低い |
| コスト | 低い | 高い |
| 実装複雑度 | 低い | 高い |
| 向いている用途 | 大量データの定期集計 | リアルタイム分析・監視 |

## マイクロバッチ（Micro-batch）

バッチとストリーミングの中間。数秒〜数分単位で小さなバッチを繰り返す。

- **Apache Spark Structured Streaming** がこの方式
- Amazon EMR や AWS Glue Streaming でも利用可能

## 試験のポイント

- **リアルタイム・低レイテンシ** → Kinesis / MSK（ストリーミング）
- **大量データの定期処理** → Glue / EMR（バッチ）
- **Kinesis Data Streams vs Firehose**
  - Streams → 独自のコンソーマーで柔軟に処理
  - Firehose → S3/Redshift等への配信が目的、管理不要
