# データレイク（Data Lake）

## 概要

あらゆる種類・形式のデータを**生のまま一か所に大量に蓄積**しておくストレージの概念。
分析する前に変換・整形する必要がなく、必要なときに必要な形で取り出して使う。

## データソース vs データレイク

混同しやすいが役割が違う。

| | 役割 | 例 |
|--|--|--|
| **データソース** | データが生まれる場所・取得元 | Aurora MySQL・Salesforce・IoTセンサー |
| **データレイク** | 複数のデータソースから集めて一元管理する場所 | Amazon S3 |

Aurora MySQL はデータソース（OLTP用のDB）であり、データレイクではない。
データレイクは**複数のソースから集約する**ことが本質。

```
Aurora MySQL（データソース①）  ─┐
Salesforceデータ（ソース②）   ─┤→ S3（データレイク）→ 分析
IoTセンサーログ（ソース③）    ─┘
```

また、**S3バケット単体がデータレイクなのではなく、S3を使ってデータレイクを構築する**イメージ。

```
【S3バケット単体】
ただのオブジェクトストレージ

【S3でデータレイクを構築】
複数ソースからの集約
+ Glue Data Catalog（スキーマ管理）
+ Athena（直接分析）
= データレイク
```

## データレイク vs データウェアハウス

| 観点 | データレイク | データウェアハウス |
|------|------------|-----------------|
| データ形式 | 何でもOK（生データ） | 構造化データのみ |
| スキーマ定義 | 読み取り時（Schema-on-Read） | 書き込み時（Schema-on-Write） |
| 対象ユーザー | データサイエンティスト・エンジニア | ビジネスアナリスト |
| コスト | 低い | 高い |
| 柔軟性 | 高い | 低い |
| クエリ速度 | 遅い（整形が必要） | 速い（最適化済み） |
| 代表AWSサービス | Amazon S3 | Amazon Redshift |

```
データレイク（S3）
├── 生ログ（JSON / CSV / Parquet）
├── 画像・動画
├── IoTセンサーデータ
└── SNSデータ
      ↓ ETL（Glue）
データウェアハウス（Redshift）
└── 整形・集計済みの構造化データ
```

## データウェアハウス vs データマート

混同しやすいが、**データマートはDWHの中に作る目的別のデータの分類**。ツールや場所ではなく概念。

```
Redshift（データウェアハウス = 入れ物）
├── schema: sales      ← 売上データマート（営業チーム向け）
├── schema: inventory  ← 在庫データマート（物流チーム向け）
└── schema: marketing  ← 行動データマート（マーケ向け）
```

| | データウェアハウス | データマート |
|--|--|--|
| 範囲 | 全社 | 特定部門 |
| データ量 | 大きい | 小さい（絞り込み済み） |
| 使う人 | データエンジニア・全社 | 特定チームのアナリスト |
| AWSサービス | Redshift（サービス自体） | Redshift内のスキーマ・テーブル |

```
「RedshiftがDWH」「その中を目的別に整理したものがデータマート」という関係。
データマートはサービスや場所の名前ではなく、データの切り出し方の概念。
```

## データレイクハウス（Data Lakehouse）

データレイクとデータウェアハウスの**いいとこ取り**をした新しいアーキテクチャ。

- データレイクの低コスト・柔軟性
- データウェアハウスのクエリ速度・信頼性

AWSでは **Amazon Redshift Spectrum** や **Apache Iceberg on S3** がこれにあたる。

## データスワンプ（Data Swamp）に注意

データレイクを適切に管理しないと「何がどこにあるかわからない沼」になる。

| 問題 | 解決策 |
|------|--------|
| データの場所がわからない | AWS Glue Data Catalog でメタデータ管理 |
| アクセス制御がない | AWS Lake Formation で権限管理 |
| データ品質が低い | AWS Glue DataBrew でプロファイリング |

## AWSにおけるデータレイク構成

```
データソース
├── RDS / Aurora        ┐
├── DynamoDB            ├─ AWS DMS / Glue → S3（データレイク）
├── SaaSアプリ          ┘       │
├── IoT / ログ → Kinesis ──────→┘
                                │
              ┌─────────────────┴──────────────────┐
              │         Amazon S3（データレイク）       │
              └──────────────┬─────────────────────┘
                             │
        ┌────────────────────┼───────────────────┐
        ▼                    ▼                   ▼
  Amazon Athena        AWS Glue ETL        Amazon EMR
（S3を直接クエリ）    （DWHへ変換）       （大規模分析）
                             ▼
                     Amazon Redshift
                    （データウェアハウス）
```

## 試験のポイント

- データレイクの中核 → **Amazon S3**
- メタデータ管理（データカタログ） → **AWS Glue Data Catalog**
- アクセス制御・権限管理 → **AWS Lake Formation**
- S3上のデータを直接SQL → **Amazon Athena**
- スキーマ定義のタイミング → レイク＝**Schema-on-Read**、DWH＝**Schema-on-Write**
