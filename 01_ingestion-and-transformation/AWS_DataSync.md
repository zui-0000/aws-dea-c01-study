# <img src="./images/aws-data-sync.png" width="36"/>&nbsp;&nbsp; AWS DataSync

## 概要

オンプレミスとAWS間、またはAWSサービス間で**データを高速・自動転送**するサービス。
手動でのデータ移行と比べて最大10倍速く転送できる。

```
オンプレミス（NFS・SMB・HDFS）
    ↓ DataSync エージェント
AWS DataSync
    ↓
AWS ストレージ（S3・EFS・FSx など）
```

---

## DataSyncのメリット

**「大量ファイル転送の面倒ごとを全部引き受けてくれる」**のが最大のメリット。

自前で実装すると大変なことを全部やってくれる：

```
【自前でやる場合の苦労】
・転送スクリプトを自分で書く
・途中で失敗したら最初からやり直し
・転送後の整合性確認スクリプトも書く
・夜間だけ転送するよう帯域制御も自前実装
・暗号化も自前対応

【DataSyncなら設定するだけ】
・途中で失敗しても自動リトライ          ✅
・転送後の整合性チェックも自動           ✅
・帯域制御も設定で対応                  ✅
・TLS暗号化がデフォルトで有効           ✅
・変更されたファイルだけ転送（増分転送） ✅
```

### 転送速度

並列処理で同時に複数ファイルを転送するため、通常のコピー（rsyncなど）より最大10倍速い。

```
通常のコピー（rsync）: ネットワーク1本で順番に転送
DataSync            : 並列処理で同時に複数転送 → 最大10倍速い
```

### 実務でイメージしやすいケース

```
「毎晩、各工場のサーバーにあるログをS3に集約したい」

自前: cronでrsyncを動かす
  → 失敗しても気づかない・監視・リトライも自前

DataSync: スケジュール設定するだけ
  → 失敗時の通知・リトライ・整合性チェックが全自動
```

## オンプレ専用ではない

参考書ではオンプレ→AWSの例が大きく書かれているが、**DataSyncはオンプレ専用ではない**。

```
オンプレ → AWS     （参考書で大きく書かれているケース）
AWS → オンプレ    （逆方向も可）
AWS → AWS         （S3 → EFS など）
他クラウド → AWS  （GCS・Azure Blob → S3）
```

DMSと並べるとどちらも「どこからでもAWSに持ってこれる」サービス。

```
              オンプレ→AWS   AWS間    他クラウド→AWS
DataSync         ✅           ✅          ✅（ファイル）
DMS              ✅           ✅          ✅（DB）
```

---

## 主な転送パターン

```
① オンプレミス → AWS（移行・バックアップ）
  NFS/SMBサーバー → S3 / EFS / FSx

② AWS → オンプレミス（配布・同期）
  S3 → オンプレミスのNFSサーバー

③ AWS → AWS（サービス間転送）
  S3 → EFS
  EFS → FSx
  S3（あるリージョン）→ S3（別リージョン）
```

---

## DataSync エージェント

オンプレミスのデータをAWSに転送する際に必要な**仮介役のソフトウェア**。
オンプレミスのサーバーにインストールして使う。

```
オンプレミスサーバー
    └── DataSync エージェント（インストール）
            ↓ インターネット or AWS Direct Connect
        AWS DataSync
            ↓
           S3 など
```

AWSサービス間の転送（S3→EFS など）はエージェント不要。

---

## 他クラウドからの転送

GCPやAzureなど他クラウドのオブジェクトストレージからもデータを転送できる。

```
Google Cloud Storage ──→ DataSync → S3 / EFS / FSx
Azure Blob Storage   ──→ DataSync → S3 / EFS / FSx
S3互換ストレージ      ──→ DataSync → S3 / EFS / FSx
```

GCSはS3互換のAPIを持っているため、DataSyncの「オブジェクトストレージロケーション」として設定することで接続できる。

```
【実務との対応】
実務でやっていたパイプライン:
Aurora → Glue → S3 → BigQuery（GCP へ）

逆方向（GCP → AWS）:
GCS → DataSync → S3（AWS に取り込む）
```

マルチクラウド環境でのデータ移行・集約にも使える。

---

## 主な機能

| 機能 | 説明 |
|------|------|
| **自動スケジュール転送** | 定期的な同期が可能 |
| **データ検証** | 転送後に整合性チェックを自動実施 |
| **暗号化** | 転送中のデータをTLSで暗号化 |
| **帯域幅制御** | 転送に使う帯域を制限できる（業務時間帯に影響させない） |
| **増分転送** | 変更されたファイルだけを転送（全件転送は不要） |

## 似たサービスとの違い

### AWS DMS との違い

| 観点 | AWS DataSync | AWS DMS |
|------|-------------|---------|
| 対象 | ファイル・オブジェクト | **データベース** |
| 用途 | ファイルシステムの移行・同期 | DBの移行・レプリケーション |
| データ形式 | NFS・SMB・S3・EFS | MySQL・PostgreSQL・Oracle など |

```
ファイルサーバーをS3に移したい → DataSync
AuroraをRedshiftに移したい    → DMS
```

### AWS Storage Gateway との違い

| 観点 | AWS DataSync | AWS Storage Gateway |
|------|-------------|-------------------|
| 主な用途 | **大量データの一括転送・移行** | オンプレとAWSのハイブリッドアクセス |
| アクセス方式 | 転送ジョブとして実行 | 常時マウントして使う |
| 向いているケース | データセンター移行・定期バックアップ | オンプレからAWSをNFSとして使い続ける |

```
「データセンターをAWSに移行したい」     → DataSync（一括転送）
「オンプレからS3をNFSとして使い続けたい」→ Storage Gateway
```

## ユースケース

| ユースケース | 説明 |
|------------|------|
| **データセンター移行** | オンプレのファイルサーバーをS3/EFSに移行 |
| **定期バックアップ** | オンプレのデータを毎晩S3にバックアップ |
| **データレイク構築** | 各拠点のデータをS3（データレイク）に集約 |
| **リージョン間レプリケーション** | S3バケットを別リージョンにコピー |

## 料金体系

**転送したデータ量（GB単位）での従量課金**。固定費・最低料金なし。

| 項目 | 内容 |
|------|------|
| 課金対象 | 転送したGB数 × 単価 |
| 固定費用 | なし |
| 最低料金 | なし |
| エージェント | 無料 |

```
✅ 増分転送（変更ファイルだけ転送）
  → 毎回全件転送しないから課金が少ない

✅ 使わないときは一切課金されない
  → スポット的な大量転送にも向いている
```

## 名前の紛らわしさに注意

「Data」とつくためDBも対象と思いがちだが、**DataSyncはファイル・オブジェクトのみ対象**。DBは対象外。

```
AWS DataSync → ファイル・オブジェクトの「同期・転送」（DBは対象外）
AWS DMS      → データベースの「移行・レプリケーション」
```

**「Sync（同期）」= ファイルシステム・オブジェクトストレージの同期**と覚える。

```
【覚え方】
ファイル・S3・EFS・FSx → DataSync
MySQL・Aurora・Oracle  → DMS
```

---

## 試験のポイント

- **ファイル・オブジェクトの転送** → DataSync（DBはDMS）
- **オンプレ → AWS への大量転送** → DataSync（エージェントが必要）
- **AWS サービス間転送** → DataSync（エージェント不要）
- **定期的な同期・増分転送** → DataSync のスケジュール機能
- **DMS との違い** → DataSync はファイル、DMS はデータベース
