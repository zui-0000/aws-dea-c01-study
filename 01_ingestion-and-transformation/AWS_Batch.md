# <img src="./images/aws-batch.webp" width="36"/>&nbsp;&nbsp; AWS Batch

## 概要

大規模な**バッチ処理ジョブを効率的に実行**するフルマネージドサービス。
Dockerコンテナ上でジョブを実行し、必要なコンピューティングリソースを自動でスケーリングする。

「大量のデータ処理・機械学習・シミュレーション」など長時間かかる処理に向いている。

---

## 構成要素

```
┌─────────────────────────────────────────┐
│              AWS Batch                  │
│                                         │
│  ジョブ定義          ジョブキュー        │
│  （何を実行するか）  （実行待ちの行列）   │
│        ↓                  ↓            │
│            コンピューティング環境        │
│         （EC2 / Fargate / Spot）        │
└─────────────────────────────────────────┘
```

| 構成要素 | 説明 |
|---------|------|
| **ジョブ定義** | 実行するDockerイメージ・CPUメモリ量・環境変数などを定義 |
| **ジョブキュー** | 実行待ちのジョブを管理する行列 |
| **コンピューティング環境** | ジョブを実行するEC2やFargateの設定 |

## Lambda・Glueとの違い

よく比較されるサービスとの違いを整理する。

| 観点 | Lambda | AWS Glue | AWS Batch |
|------|--------|----------|-----------|
| 実行時間上限 | **15分** | 48時間 | **無制限** |
| 用途 | 軽量・イベント処理 | ETL処理 | 大規模・長時間バッチ |
| コンテナ | 非対応 | 非対応 | **Docker対応** |
| コスト | 実行時間で課金 | DPU時間で課金 | EC2/Fargate時間で課金 |
| 向いている処理 | 数秒の処理 | データ変換・ETL | 数時間〜数日の処理 |

```
15分以内に終わる処理       → Lambda
ETL・データ変換           → AWS Glue
それ以上・Dockerが必要     → AWS Batch
```

## スポットインスタンスとの組み合わせ

AWS Batchはスポットインスタンス（通常の最大90%オフ）を使えるのが強み。

```
コンピューティング環境
├── オンデマンドインスタンス（高コスト・安定）
└── スポットインスタンス（低コスト・中断リスクあり）
            ↑
    バッチ処理は中断されても再試行できるのでスポットと相性が良い
```

## ECSスケジュールタスクとの比較

どちらもバッチ処理に使えるが、ジョブの複雑さで使い分ける。

| 観点 | ECSスケジュールタスク | AWS Batch |
|------|-----------------|-----------|
| ジョブキュー | なし | あり（順番・優先度管理） |
| ジョブの依存関係 | なし | あり（AがないとBを実行しない） |
| スポット最適化 | 自分で実装 | 自動（中断時の再試行も自動） |
| リソース管理 | 常にクラスターが必要 | 必要な時だけEC2を起動・終了 |
| 並列実行管理 | 自分で制御 | 自動でスケーリング |
| 向いている用途 | シンプルな定期実行 | 大量・複雑なジョブ管理 |

```
「毎朝9時にレポートを生成する」
→ シンプルな定期実行 → ECSスケジュールタスクで十分

「1000個のファイルを並列変換して、全部終わったら集計する」
→ ジョブ管理が複雑 → AWS Batchが向いている
```

**判断軸：「ジョブが1つで定期実行するだけ」→ ECS、「複数ジョブ・並列・依存関係あり」→ AWS Batch**

## Step Functionsとの組み合わせ

複数のバッチジョブをワークフローとして管理できる。
成功・失敗に応じたSlack通知や、EventBridgeでのスケジュール起動も組み合わせ可能。

```
EventBridge（毎日0時にスケジュール起動）
    ↓
Step Functions（ワークフロー全体を管理）
    ├── AWS Batch（データ処理）
    │     ↓ 成功
    ├── AWS Batch（次の処理）
    │     ↓ 失敗
    │   Lambda（Slack通知「❌ 失敗しました」）
    │     ↓ 成功
    └── Lambda（Slack通知「✅ 完了しました」）
```

「処理の主役」が変わるだけで、EventBridge → Step Functions の構成パターンはGlue・Lambdaと同じ。

### Step Functions の Map State との比較

実務で Step Functions Map + Lambda で並列処理を実装している場合、AWS Batch はその大規模版にあたる。

| 観点 | Step Functions Map + Lambda | AWS Batch アレイジョブ |
|------|----------------------------|----------------------|
| 並列管理 | Step Functionsが管理 | Batchが管理 |
| 時間制限 | 15分（Lambda制限） | なし |
| 向いている件数 | 数十〜数百件 | 数千〜数万件 |
| コスト | Lambda課金 | EC2/Spot課金 |

### アレイジョブ（Array Jobs）

AWS Batch独自の並列実行機能。同じ処理をN並列で実行する場合はMapが不要になる。

```
【Step Functions Map + Lambda（従来）】
Step Functions
    ↓ Map State
    ├── Lambda（ファイル1）
    ├── Lambda（ファイル2）
    └── Lambda（ファイル1000）← Step Functionsが並列管理

【AWS Batch アレイジョブ】
Step Functions
    ↓ Submit Job（1回だけ）
    AWS Batch（1000並列）
    ├── ジョブ1
    ├── ジョブ2
    └── ジョブ1000 ← Batchが並列管理
```

**MapとBatchの使い分け：**

```
同じ処理を大量並列実行したい → Batchのアレイジョブで完結（Map不要）
種類が違う処理を並列実行したい → Step Functions Map → 各ブランチでBatchジョブを投げる
```

## 試験のポイント

- **実行時間に上限がない** → 長時間処理はAWS Batch（Lambdaの15分制限を超える場合）
- **Dockerコンテナで実行** → カスタム環境が必要な場合に強い
- **スポットインスタンスでコスト削減** → バッチ処理との相性が良い
- **GlueとBatchの使い分け** → ETL変換はGlue、それ以外の長時間バッチはBatch
- **Step Functionsと組み合わせ** → 複数ジョブのワークフロー管理
