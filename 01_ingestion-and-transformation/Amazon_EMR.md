# <img src="./images/aws-amazon-emr.png" width="36"/>&nbsp;&nbsp; Amazon EMR（Elastic MapReduce）

## 概要

**オープンソースのビッグデータフレームワークをマネージドクラスターで実行**するサービス。
Apache Spark・Hadoop・Hive・Prestoなど多様なフレームワークを使える。

```
データソース（S3・RDS・DynamoDBなど）
        ↓
Amazon EMR（クラスター上でSparkなどを実行）
        ↓
Amazon S3 / Redshift / DynamoDB（結果の保存）
```

AWS Glue もSparkベースだが、EMR はより細かい制御と柔軟性が必要な大規模処理向け。

---

## 対応フレームワーク

| フレームワーク | 用途 |
|-------------|------|
| **Apache Spark** | 大規模データ処理・ML・ストリーミング |
| **Apache Hadoop（MapReduce）** | 分散バッチ処理 |
| **Apache Hive** | SQLライクなデータウェアハウス処理 |
| **Apache HBase** | NoSQL列指向DB（大規模テーブル） |
| **Presto / Trino** | インタラクティブな高速SQLクエリ |
| **Apache Flink** | ストリーミング処理 |
| **Apache Pig** | データフロー言語でのETL |

---

## クラスターの構成

EMRはノードの集合（クラスター）で動く。ノードは3種類ある。

```
EMR クラスター
├── Primary Node（マスターノード）× 1台
│     クラスター全体の管理・調整
│     タスクの割り振り
│
├── Core Node（コアノード）× 複数台
│     データの保存（HDFS）
│     タスクの実行
│     ※ 削除するとデータ損失リスクあり → On-Demand 推奨
│
└── Task Node（タスクノード）× 複数台（オプション）
      タスクの実行のみ（データ保存なし）
      削除してもデータ損失なし → Spot インスタンス活用可
```

---

## ストレージの選択肢

| ストレージ | 説明 | 特徴 |
|----------|------|------|
| **EMRFS（S3）** | S3をストレージとして使う | クラスター終了後もデータ残る・最もよく使われる |
| **HDFS** | クラスター内の分散ストレージ | 高速だがクラスター終了でデータ消える（一時的な中間データ向け） |
| **ローカルファイルシステム** | 各ノードのディスク | 一時的な処理のみ |

```
本番データの永続化 → EMRFS（S3）
処理中の中間データ → HDFS（高速・一時的）
```

### EMRFS の詳細

HadoopやSparkは元々 HDFS という独自のファイルシステムで動く設計になっている。
S3はHDFSではないため、そのままでは読めない。

**EMRFS = EMRとS3をつなぐファイルシステムの通訳レイヤー。**

```
Hadoop / Spark のコード
    ↓
「s3://bucket/data を読んで」という命令
    ↓
EMRFS（S3をHDFSのように見せる通訳）
    ↓
Amazon S3（実際のデータはここ）
```

EMRFSがあることで、SparkのコードはS3を「普通のファイルシステム」として扱える。

**HDFSとEMRFSの決定的な違い：**

```
【HDFS】
データはクラスターのEC2ディスク上に存在
    ↓ クラスターを終了
データも消える → クラスターを動かし続けないといけない → コスト増大

【EMRFS（S3）】
データはS3上に存在（クラスターの外）
    ↓ クラスターを終了
データはS3に残る ✅ → 必要なときだけクラスターを起動すればいい
```

これが**コンピュートとストレージの分離**という考え方。

```
従来（HDFS）:
クラスター = 計算 + データ保管
→ データを消したくないからクラスターを止められない

EMRFS（S3）:
クラスター = 計算だけ
S3        = データ保管だけ
→ 計算が終わったらクラスターを終了できる → 一時クラスターが成立する！
```

## クラスターの種類

### 一時クラスター（Transient Cluster）

処理が終わったらクラスターを自動終了する。

```
クラスター起動 → 処理実行 → 自動終了
→ 使った時間だけ課金 → コスト効率が高い
→ 定期バッチ処理に向いている
```

### 常時起動クラスター（Long-running Cluster）

クラスターを起動したままにする。

```
クラスター起動 → ジョブを随時投入 → 常時稼働
→ 起動時間分ずっと課金
→ インタラクティブな分析・頻繁にジョブを投入する場合に向いている
```

---

## コスト最適化：Spot インスタンスの活用

Task Node はデータを持たないため、中断されてもデータ損失がない。
→ **Spot インスタンス**（余剰リソース・安い）を使うことでコストを大幅削減できる。

```
Primary Node  → On-Demand（中断不可）
Core Node     → On-Demand（データ損失リスクあるため）
Task Node     → Spot インスタンス（安い・中断されてもOK）
```

---

## ステップ（Steps）

EMR クラスターに投入する作業の単位。

```
Step 1: S3からデータを読み込む
Step 2: Sparkで変換・集計する
Step 3: 結果をS3に書き出す
```

Step Functions と組み合わせて複数Stepを順番に実行・エラー制御することも多い。

## Bootstrap Actions

クラスター起動時にノード上で自動実行されるスクリプト。
Hadoop が起動する前に実行され、ソフトウェアのインストールや設定変更に使う。

```
【Bootstrap Actions の使いどころ】
・追加ライブラリのインストール（pip install など）
・設定ファイルの変更
・カスタムスクリプトの配置
```

## EMR Serverless

クラスター管理不要でEMRワークロードを実行できるデプロイ形式の1つ（2022年6月にGA）。
GlueのようにサーバーレスでSparkやHiveを使える。

```
EMR（通常）      → クラスター設計・ノード数・インスタンスタイプを自分で決める
EMR Serverless → インフラ管理不要。Glueに近い使い勝手
```

## デプロイ形式

EMR は4つのデプロイ形式に対応している。

| | 実行場所 | 向いているケース |
|--|--|--|
| **EMR on EC2** | AWSクラウド | 従来型・細かい制御・オンプレ移行 |
| **EMR on EKS** | AWSクラウド（K8s） | Kubernetesで統一管理 |
| **EMR Serverless** | AWSクラウド | 手軽・サーバーレス |
| **EMR on Outposts** | オンプレミス | データをクラウドに出せない・法規制対応 |

公式が提供するデプロイ形式は EC2 / EKS / Serverless / Outposts の4つで、ECS は含まれない。Spark は Kubernetes（EKS）をネイティブサポートしているため EKS が採用されている。

### EMR on AWS Outposts

**AWSのサーバーラックを物理的に自社データセンターに設置**し、その上でEMRを動かす形式。
「ソフトウェアをインストールする」のではなく、AWS が実際のハードウェアを送り込んでくる。

```
【通常のAWS】
AWSのデータセンター
    └── AWSのサーバーラック → EMRが動く

【AWS Outposts】
自社のデータセンター
    └── AWSが設置したサーバーラック → EMRが動く
         ↑ 物理的にはオンプレにある
         ↑ 管理・運用はAWSが担当（コントロールプレーンはAWS側）
```

**データの出力先：**

```
EMR on Outposts の出力先
├── オンプレのストレージ  ← データをクラウドに出さずに完結
├── Outposts上のストレージ ← 同上
└── S3（クラウド）        ← クラウドに出してもよい場合

法規制で「データをクラウドに出せない」→ オンプレストレージに出力
```

**使うケース：**

```
「医療データは院内サーバーに置かなければならない」
「金融データは国外に出せない（データレジデンシー要件）」
→ でも EMR の機能は使いたい → EMR on Outposts
```

## EMR が必要になるケース

標準的なETLなら Glue で十分なことが多い。EMR を選ぶ理由は以下の4つ。

### ① Glue が対応していないフレームワークが必要

```
Glue が使えるのは Spark と Python Shell のみ

「HiveQL でSQLを書いて処理したい」
「Presto で大量データをインタラクティブに高速クエリしたい」
「HBase（列指向NoSQL）を使いたい」
→ Glue では対応不可 → EMR が必要
```

### ② オンプレの Hadoop 資産をそのまま移行したい

```
オンプレで長年動いてきた Hadoop / Hive ジョブがある
→ Glue に書き直すコストが膨大
→ EMR にそのまま持ってくる（リフト＆シフト）ほうが現実的
```

### ③ クラスターを細かく制御したい

```
「GPU インスタンスで Spark ML 処理をしたい」
「メモリ最適化インスタンスを指定したい」
「特定の Spark バージョン・設定にしたい」
→ Glue は DPU 単位で抽象化されていて細かい制御ができない
→ EMR ならインスタンスタイプを自由に指定できる
```

### ④ 超大規模でコストを極限まで下げたい

```
常時稼働クラスター → Reserved Instance で大幅割引
Task Node → Spot Instance でさらにコスト削減
→ PB規模・常時稼働なら Glue の DPU 課金より安くなるケースがある
```

```
【Glue で十分なケース】
├── S3 → 変換 → Redshift の標準ETL
├── Python で書ける処理
├── スケジュールバッチ
└── サーバー管理したくない

【EMR が必要なケース】
├── Hive・Presto・HBase を使いたい
├── オンプレ Hadoop からの移行（リフト＆シフト）
├── GPU インスタンスで ML 処理
└── PB規模で Reserved Instance でコスト最適化
```

## Glue との比較（最重要）

| 観点 | Amazon EMR | AWS Glue |
|------|-----------|---------|
| サーバー管理 | 必要（クラスター設計が必要） | 不要（サーバーレス） |
| 対応フレームワーク | Spark・Hadoop・Hive・Prestoなど多数 | Spark・Python Shell のみ |
| 自由度 | 高い（細かい設定が可能） | 低い（マネージドな範囲内） |
| 使いやすさ | 難しい | 簡単 |
| コスト | クラスター稼働時間課金 | DPU時間課金 |
| 向いているケース | 複雑な大規模処理・複数フレームワーク併用 | 標準的なETL |

```
標準的なETLパイプラインを手軽に → AWS Glue
複雑な処理・Hive/Presto/HBaseなど複数フレームワークが必要 → Amazon EMR
```

## Glue Data Catalog との連携

EMR は Glue Data Catalog をメタストアとして使える。

```
Glue Crawler がS3をスキャン → Data Catalog にテーブル登録
        ↓
EMR（Hive / Spark）がData Catalogを参照してクエリ
```

Athena・Glue ETL・EMR が同じData Catalogを共有できる。

## 試験のポイント

- **マネージドクラスターで複数フレームワークを動かす** → Amazon EMR
- **Glueとの違い** → EMRはクラスター管理が必要・自由度が高い
- **Task Node に Spot インスタンス** → コスト削減（データ損失リスクなし）
- **Core Node は On-Demand 推奨** → 削除するとデータ損失リスクあり
- **EMRFS（S3）** → クラスター終了後もデータが残る永続ストレージ
- **一時クラスター** → バッチ処理後に自動終了してコスト削減
- **Bootstrap Actions** → クラスター起動時にカスタムスクリプトを実行
- **Glue Data Catalog を共有** → Athena・Glue・EMRで同じメタデータを参照
